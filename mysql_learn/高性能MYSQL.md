##### 锁

- 共享锁（读锁）：多个客服可以同时读一个资源互不影响。
- 排他锁（写锁）：一个写锁会阻塞其他的读锁和写锁。
- 锁粒度：锁定的数据越少，系统的并发程度越高。
- 表锁：锁定整张表。开销最小的策略。
- 行级锁：最大程度支持并发，开销最大的策略。

# 事务

##### 隔离级别

##### 死锁

- 两个或多个事务在同一个资源上互相占用，并请求锁定对方占用的资源。事务都等待对方释放锁， 同时又持有对方需要的锁。
- 数据库实现可死锁检测和死锁超时机制。比如InnoDB处理方式是将持有最少行级排他锁的事务回滚。
- 程序不需要考虑死锁，只重新执行因死锁回滚的数据即可。

##### 事务日志

- 可以提高事务的执行效率。
- 存储引擎修改表数据时修改内存拷贝，再把修改行为记录到硬盘上的事务日志中，而不是每次将修改的数据本身持久到硬盘。事务日志采用追加的方式，因此写日志操作实在磁盘上一小块区域顺序I/O操作，而不像随机I/O操作在磁盘的多个地方移动磁头。事务日志持久化以后内存中修改的数据可以后台慢慢写到磁盘上。修改数据一共需要写两次磁盘。
- 如果修改的数据已经记录到事务日志并持久化，但数据本身没写入磁盘，此时系统崩溃，存储引擎能在重启是自动恢复这些数据。

##### 事务中混合使用存储引擎

如果在事物中使用了事务和非事务的表，正常提交的情况没问题，如果事物需改回滚非事务的表数据不能撤销，造成数据不一致。

##### 隐式和显式锁

- 隐式
  - 事务执行过程中随时都可以执行锁定，只有执行到commit或roll back时才会释放锁，并且同时释放。
  - InnoDB会根据隔离级别自动加锁。

- 显式 lock tables 
  - lock tables 和事务互相影响。最好不要使用。

# 存储引擎

##### InnoDB

- 事务引擎。
- 行级锁。

##### MyISAM

- 没有事务。
- 对整张表加锁，而不是针对行。读取时对所有用到的表加共享锁，写入时对表加排他锁。但在表有读取查询的同时也可以往表中插入新的记录（称为并发插入）。

##### Memory

- 适用于需要快速访问数据，并且这些数据不会被修改，重启后丢失数据也没关系。
- 比MyISAM快一个数量级，数据保存在内存中，不需要进行磁盘I/O。表结构重启后会保留但数据丢失。
- 表级锁，并发写入性能低。不支持BOOL或TEXT类型的列。
- 查询过程中需要临时表保存中间结果，可以使用Memory。如果中间结果太大超过Memory表限，或含有BOOL，TEXT类型字段，临时表会转换为MyISAM。
- 临时表：指使用create temporary table语句创建的表，可以使用任何存储引擎。临时表只在单个连接中可见，断开连接是临时表也不存在。

##### Archive

- 仅支持INSERT和SELECT操作。Archive会缓存所有的写并用zlib对插入行进行压缩，所以比MyISAM磁盘I/O更少。插入非常快。
- 每次执行SELECT会执行全表扫描。
- 支持行级锁和专用的缓冲区，实现高并发插入。批量插入完成之前对读操作是不可见的。
- SELECT开始查询直到返回表中所有存在的行数之前，会阻止其他SELECT的执行，以实现一致性读。
- 适用于日志和数据采集的应用，这类应用做数据分析时往往需要扫描全表。或需要快速INSERT操作的场景。

# 创建高性能索引

先在索引文件中找到对应值，然后根据匹配的索引记录找到对应的数据行，不在需要进行全表扫描。

叶子节点的指针指向被索引的数据，而不是其他节点。

索引可以包含该一列或多列的值，如果索引包含多列值顺序很重要，MySQL只能高效的使用索引的最左前缀列。依据create table定义索引时的顺序。

##### 索引好处

- 快速定位到表中的指定位置
- B-tree使用排序存储，所以MySQL可以使用order by 和 group by 
- 因为索引中存储了实际的列值，某些查询只使用索引就能完成查询。
- 减少了扫描的数据量
- 帮助服务器避免排序和临时表
- 将随机I/O变为顺序I/O

##### 索引有效的类型

- 全值匹配：与索引中的所有列匹配
- 匹配最左前缀：只使用索引的第一列
- 匹配列前缀：值匹配某一列的开头部分
- 匹配范围值
- 某一列索引精确匹配，另一列范围匹配
- 只访问索引列的查询：只访问索引，不访问数据行。

##### 索引并不总是最好的

- 非常小的表全表扫描更高效
- 中大型的表索引非常有效
- 特大型的表建立和使用索引代价随之增大



独立的列：索引不能表达式的一部分，也不能是函数的参数，否则无法使用索引。

```
select person_id from user where person_id + 1 = 30;
```

##### 覆盖索引

- 如果一个索引包含（覆盖）了所需要查询的字段值，称为覆盖索引。
- 可以使用索引直接获取数据，而不用需要读取数据行。极大提高性能。



# 优化查询

查询优化、索引优化、库表结构优化

##### 是否向数据库请求了不需要的数据

- 查询不需要的记录：如果查询到100条，但只需要其中10条，查询后加上LIMIT
- 取出全部的列：select * 为服务器带来额外的I/O、内存、CPU消耗。
- 重复查询相同的数据：可以将结果缓存

##### 是否扫描额外的记录

三个指标：响应时间、扫描行数、返回行数。

- 响应时间：

  - 服务时间：真正执行查询花费的时间
  - 队列时间：服务器等待资源花费的时间。I/O、行级锁等。

- 扫描行数优化方法：

  - 使用覆盖索引，把需要的列放在索引里面。
